## 0) الوضع الحالي (مراجعة المشروع)

### ما هو جاهز في الكود
- واجهة React 19 + Vite + TypeScript + Tailwind v4 (RTL-first).
- Backend: Vercel Serverless (`api/*`) مع middleware صلاحيات و JWT.
- تكامل Supabase: Auth + Postgres + Storage.
- سكربتات SQL مرحلية: P0 أمن/RLS إضافي، P1 ورديات/مرتجعات/جرد…، P2 ولاء/أسعار/BOM/AI.
- Google OAuth (عبر بروكسي Design Arena حالياً).
- خطافات Sentry جاهزة عند ضبط DSN.
- WhatsApp: deeplink منفّذ؛ Cloud API اختياري.
- Web Push (VAPID) اختياري.
- Offline: outbox + Service Worker.
- نشر تجريبي سابق عبر بيئة Design Arena / Vercel preview.

### ما **ليس** جاهزاً كبنية تحتية “ملكك أنت”
| العنصر | الحالة الحالية | المخاطرة |
|--------|----------------|----------|
| مشروع Supabase | مربوط بمشروع **مشترك/مؤقت** لبيئة البناء (`FULLSTACK_PROJECT_REF`) | فقدان بيانات، مشاركة مفاتيح، لا SLA ملكك |
| المفاتيح في `.env` / `vercel.json` | موجودة لبيئة الـ sandbox | **لا تُعامل كمفاتيح إنتاج نهائية** |
| GitHub repo خاص | غير مؤكد كمستودع ملكية للمنتج | لا CI/CD ولا سجل تغييرات رسمي |
| Vercel production خاص | نشر sandbox/preview وليس بالضرورة production brand | نطاق ودومين وفواتير منفصلة |
| تشغيل SQL على “إنتاجك” | README يصرّح: تشغيل `p2` على كل إنتاج ما زال ☐ | جداول P1/P2 أو فهارس/RLS ناقصة = أعطال صامتة |
| أسرار اختيارية (Sentry/VAPID/WhatsApp Cloud) | غير إلزامية للتشغيل الأساسي | مراقبة/إشعارات ناقصة |

> **خلاصة:** الكود ناضج لمرحلة **تجربة قبول (Pilot)**، لكن البنية التحتية “المملوكة” يجب فصلها عن sandbox قبل أي عميل حقيقي.

## 1) هل يجب إنشاء مشروع جديد على Supabase الآن؟

### القرار: **نعم — الآن** (قبل أي عميل تجريبي حقيقي وقبل P3)

### السبب
1. المشروع الحالي على Supabase في بيئة التطوير/الـ sandbox **ليس** قاعدة بيانات إنتاج مملوكة بالكامل لعلامتك.
2. README يشترط صراحة تشغيل سكربتات SQL على بيئة الإنتاج.
3. البيانات التجريبية (منتجات، مبيعات، مستأجرون) مختلطة مع تطوير؛ تحتاج **بداية نظيفة** أو على الأقل مشروع Dev منفصل.
4. تغيير `project ref` لاحقاً يفرض إعادة ربط Auth redirect URLs، Storage buckets، والمفاتيح في Vercel = إعادة عمل.
5. Service Role مستخدم من الـ API؛ امتلاك المشروع يعني التحكم في التدوير (rotation) والصلاحيات والنسخ الاحتياطي.

### متى يمكن التأجيل؟
- فقط إذا كان الهدف **عرض UI داخلي** بدون بيانات حقيقية وبدون مستخدمين خارجيين.
- **لا تؤجّل** إذا اقترب “أول متجر تجريبي” أو “أول نسخة تجريبية للعملاء”.

## 2) اسم المشروع المقترح — وهل نحتاج Dev و Production؟

### التسمية المقترحة

| البيئة | اسم مشروع Supabase المقترح | Region مقترحة |
|--------|------------------------------|---------------|
| **Production** | `rafd-prod` | أقرب للمستخدمين: `eu-central-1` أو `ap-southeast-1` (قيّم latence لليمن/السعودية؛ غالباً Europe أفضل من US) |
| **Development / Staging** | `rafd-dev` | نفس الـ region لتقليل فروق السلوك |

أسماء بديلة مقبولة:
- `rafd-ye-prod` / `rafd-ye-dev` (إن أردت تمييز السوق).
- Organization: `rafd-hq` على Supabase.

### هل تحتاج مشروعين؟

| السيناريو | التوصية |
|-----------|---------|
| أول Beta لـ 1–3 متاجر خلال أسابيع | **نعم: Dev + Prod** (الحد الأدنى الصحي) |
| ميزانية/وقت شديد الضيق جداً | ابدأ بـ **Prod فقط** + فرع Git `staging` يستخدم نفس المشروع مع schema prefix **غير مفضل** |
| بعد أول 10 عملاء | أضف **Staging** ثالث اختياري (`rafd-staging`) إن لزم |

### السبب لاختيار Dev + Prod
- تجربة SQL/RLS/البذور دون كسر بيانات متاجر حقيقية.
- Auth users في Dev لا يتسربون لإنتاج.
- Storage (شعارات/صور) منفصل.
- اختبار Google OAuth redirect على دومين preview دون لمس إنتاج.

### إعدادات يُنصح بها عند الإنشاء
- تفعيل **Point-in-Time Recovery** على Prod (خطة Pro إن أمكن بعد أول عميل دافع).
- كلمة مرور DB قوية + عدم مشاركة Service Role.
- تعطيل signup العام لاحقاً إن كان onboarding بدعوة فقط (قرار منتج).
- إنشاء bucket عام للوسائط: نفس اسم الكود الحالي إن وُجد (`rafd-media` أو ما يعادله في `api/upload.js`).

## 3) هل يجب تنفيذ جميع ملفات SQL (P0 و P1 و P2) الآن أم لاحقاً؟

### القرار: **نفّذ الثلاثة الآن — بالترتيب — على Dev فوراً، وعلى Prod قبل أول Beta**

الترتيب الإلزامي (كما في README §17):

```text
1) supabase/p0_security.sql
2) supabase/p1_features.sql
3) supabase/p2_features.sql
```

### ماذا يفعل كل ملف (ملخص)

| الملف | الغرض | إن لم يُنفَّذ |
|-------|--------|----------------|
| **P0** | أعمدة ضريبة/idempotency للمبيعات، وزن `weight_g`، أعمدة ضريبة المستأجر، تفعيل RLS على جداول حساسة | فشل offline idempotency، وزن الخضار، فجوات أمن |
| **P1** | ورديات، مرتجعات، جرد، دعوات، أجهزة دفع، اشتراكات منصة، إعلانات… | شاشات P1/API ترجع 500 أو جداول فارغة مكسورة |
| **P2** | ولاء، قوائم أسعار، وصفات/BOM، محادثات AI | ولاء/تسعير/تصنيع/AI غير قابلة للتشغيل |

### السبب لتنفيذها **الآن** وليس بعد P3
1. الكود **يعتمدها بالفعل** (APIs موجودة).
2. تأجيل SQL بعد ملء بيانات حقيقية يزيد تعقيد الـ migrations (تضارب أعمدة/فهارس).
3. README يعتبر تشغيل SQL على الإنتاج بند إطلاق مفتوح ☐.
4. P3 (جودة هندسية) **لا يستبدل** الـ schema؛ يؤجّل SQL فقط يؤخر الإطلاق دون فائدة.

### ملاحظات تنفيذ
- شغّل عبر **SQL Editor** في Supabase (أو CI migration لاحقاً).
- احفظ لقطة (dump) قبل/بعد.
- بعد التنفيذ: تحقق بـ `npm test` محلياً + smoke على `/api/health` إن وُجد.
- البذور (seed): Dev نعم ببيانات عربية تجريبية؛ Prod **حد أدنى** (خطط اشتراك، مستخدم سوبر أدمن، لا فواتير وهمية لعملاء حقيقيين).

## 4) هل يجب ربط مفاتيح Supabase الآن أم بعد مراحل معيّنة؟

### القرار: **اربط المفاتيح الآن — فور إنشاء مشروعك — قبل Beta**

### المفاتيح المطلوبة

| المفتاح | أين | ملاحظات |
|---------|-----|---------|
| `VITE_SUPABASE_URL` | Frontend (Vercel + `.env.local`) | عام |
| `VITE_SUPABASE_ANON_KEY` | Frontend | عام مع RLS |
| `NEXT_PUBLIC_SUPABASE_URL` | API serverless | يطابق URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | إن لزم | حسب `db-client`/أنماط المشروع |
| `SUPABASE_SERVICE_ROLE_KEY` | **Server only** (Vercel env) | **لا تُعرض للمتصفح أبداً** |

### ترتيب الربط
1. أنشئ `rafd-dev` → انسخ المفاتيح → ضعها في `.env.local` للتطوير.
2. نفّذ SQL على Dev → اختبر login + POS + upload.
3. أنشئ `rafd-prod` → ضع المفاتيح في **Vercel Production env**.
4. نفّذ SQL على Prod → deploy.

### لماذا ليس “بعد انتهاء ميزات أخرى”؟
- تأخير الربط يعني استمرار الاعتماد على sandbox.
- OAuth redirect URLs و Site URL في Supabase Auth يجب ضبطها على دومينك النهائي مبكراً.
- تدوير المفاتيح لاحقاً يكسر الجلسات النشطة.

### أمان
- لا ترفع Service Role إلى Git.
- افصل env: Preview → Dev keys؛ Production → Prod keys.
- بعد النقل من Design Arena: ألغِ الاعتماد على `FULLSTACK_RESTORE_*` في مسارك الإنتاجي إن لم تعد تحتاجها.

## 5) هل يجب إنشاء مشروع GitHub الآن وربطه؟

### القرار: **نعم — الآن (يوم البنية التحتية)**

### السبب
1. الكود ضخم (P0–P2)؛ بدون Git رسمي لا يوجد مصدر حقيقة (source of truth).
2. مطلوب لربط Vercel CI/CD ومراجعات PR قبل Prod.
3. يحمي من فقدان بيئة الـ sandbox.
4. يفعّل لاحقاً: Dependabot، أسرار GitHub Actions، حماية فرع `main`.

### الهيكل المقترح للمستودع
```text
rafd-app                 # monorepo الحالي (Vite + api/)
├── main                 # إنتاج فقط عبر PR
├── develop              # دمج يومي
└── release/x.y.z        # اختياري للتجريبيات
```

### إعدادات GitHub الموصى بها فوراً
- Private repository.
- Branch protection على `main`: PR + status checks (`npm test`, `npm run build`).
- Secrets **لا** تُخزَّن في repo؛ فقط في Vercel/GitHub Actions.
- `.gitignore` يستمر بتجاهل `.env`, `node_modules`, `dist`, `.vercel`.

### ماذا تربط؟
- اربط جهاز التطوير + Vercel + (لاحقاً) أي CI.
- لا حاجة لـ GitHub Teams المتقدمة الآن.

## 6) هل يجب إنشاء مشروع Vercel الآن وربطه بـ GitHub؟

### القرار: **نعم — الآن**

### السبب
1. التطبيق مصمم أصلاً كـ **Vite SPA + Vercel Serverless `api/`**.
2. النشر اليدوي من sandbox لا يغني عن pipeline إنتاج.
3. ربط GitHub → Vercel يعطيك:
   - Preview لكل PR (على `rafd-dev` keys)
   - Production على `main` (على `rafd-prod` keys)

### إعدادات Vercel
| إعداد | قيمة مقترحة |
|--------|-------------|
| Framework preset | Vite |
| Build | `npm run build` |
| Output | `dist` |
| Install | `npm ci` أو `npm install` |
| Node | 20.x LTS |
| Env Preview | مفاتيح `rafd-dev` |
| Env Production | مفاتيح `rafd-prod` |

### ما الذي لا تفعله
- لا تعتمد `vercel.json` المعبأ بمفاتيح sandbox كمرجع أسرار نهائي؛ انقل الأسرار إلى **Vercel Dashboard → Environment Variables**.
- لا تضع Service Role في متغيرات `VITE_*`.

### الدومين
- مؤقتاً: `*.vercel.app`
- قبل عملاء جادين: `app.rafd.sa` أو `app.rafd.ye` / `pos.rafd.app` + SSL تلقائي

## 7) متى تُنشر أول نسخة تجريبية؟ وهل بعد Vercel أم بعد مرحلة معيّنة؟

### القرار: **أول Beta بعد إغلاق “مسار البنية التحتية” وليس بعد P3**

P3 (جودة هندسية: e2e، OpenAPI، code-splitting، خطوط PDF عربية كاملة) **تحسينات**؛ README يعتبر المنتج قابلاً لتجربة قبول ميدانية بعد P0–P2 **بشرط** SQL + أسرار + أجهزة حقيقية.

### تعريف “أول نسخة تجريبية” (Beta-0)

| البند | مطلوب؟ |
|-------|--------|
| Supabase Prod + SQL P0→P2 | ✅ |
| Vercel Production مربوط بـ GitHub `main` | ✅ |
| تسجيل دخول demo + owner حقيقي | ✅ |
| POS: بيع نقدي + باركود + فاتورة | ✅ |
| مخزون كرتون/حبة (+ وزن خضار إن لزم للتجربة) | ✅ |
| مستأجر واحد معزول | ✅ |
| Sentry | مُستحسن جداً |
| WhatsApp Cloud / FCM / R2 | ❌ ليس لـ Beta-0 |
| P3 كامل | ❌ لا تنتظره |

### التوقيت النسبي
```text
[اليوم] خطة بنية
   → GitHub
   → Supabase Dev + SQL
   → ربط مفاتيح محلياً + smoke
   → Vercel + GitHub (Preview)
   → Supabase Prod + SQL
   → Vercel Production deploy
   → اختبار قبول داخلي 48–72 ساعة
   → Beta مع متجر واحد (Pilot)
   → (لاحقاً) P3 موافقة منفصلة
```

**أول deploy على Vercel Production = مرشح Beta** فقط بعد smoke checklist (القسم 11).

## 8) ما الخدمات الخارجية التي يجب إنشاؤها الآن؟

### أ) أنشئ / فعّل **الآن** (يوم البنية)

| الخدمة | القرار | السبب |
|--------|--------|--------|
| **Supabase (Dev + Prod)** | إلزامي الآن | DB + Auth + Storage عمود فقري |
| **GitHub** | إلزامي الآن | ملكية الكود + CI |
| **Vercel** | إلزامي الآن | استضافة مطابقة للعمارة |
| **Google Cloud OAuth Client** (خاص بك) | مُستحسن الآن / إلزامي قبل عملاء يعتمدون Google login | الحالي عبر `VITE_GOOGLE_AUTH_PROXY` لـ Design Arena؛ للإنتاج تحتاج OAuth client + redirect على دومينك |
| **Supabase Storage bucket** | الآن مع SQL/الإعداد | شعارات المتجر وصور المنتجات |
| **Sentry** (مشروع واحد `rafd-web`) | مُستحسن جداً الآن | التقاط أخطاء Beta؛ الكود جاهز لـ DSN |
| **DNS / Domain** | مُستحسن الآن إن وُجد دومين | تقليل إعادة ضبط Auth URLs لاحقاً |

### ب) جهّز حساباً لكن يمكن تأجيل التفعيل الكامل

| الخدمة | القرار | السبب |
|--------|--------|--------|
| **WhatsApp Cloud API (Meta)** | اختياري لـ Beta-0؛ إلزامي إن وعد المبيعات “إرسال فاتورة رسمي من رقم أعمال” | deeplink `wa.me` يكفي للتجربة؛ Cloud للقالب/الإنتاج |
| **Web Push VAPID** (أو خدمة Push) | اختياري | يعمل بدونها؛ مفيد لمدير المتجر |
| **Resend أو Brevo** | أجّل | لا يوجد مسار بريد معاملات حرج مكتمل كاعتماد إطلاق؛ الدعوات قد تكون برابط داخل التطبيق |
| **Firebase Cloud Messaging** | أجّل | لا يوجد تطبيق متجر أصلي منفصل كقناة إلزامية؛ `/mobile` PWA أولاً |
| **Cloudflare R2 أو S3** | أجّل | Supabase Storage يكفي حتى حجم وسائط ملحوظ |
| **Stripe / مزود دفع اشتراكات** | أجّل أو يدوياً | إدارة اشتراك داخلية موجودة؛ بوابة دفع حقيقية لاحقاً |
| **Posthog/Analytics** | اختياري | بعد Pilot لفهم الاستخدام |
| **Uptime monitor** (BetterUptime/Checkly) | مُستحسن خفيف | على `/api/health` |

### ج) لا تُنشأ الآن
- Kubernetes / خوادم VPS مخصصة (لا حاجة — Serverless كافٍ).
- CDN منفصل معقّد (Vercel كافٍ في البداية).
- خدمة SMS OTP منفصلة (إلا إذا قررت استبدال البريد).

## 9) ما الخدمات الإلزامية قبل إطلاق أول عميل (Pilot حقيقي)؟

قائمة **الحد الأدنى غير القابل للتفاوض**:

1. **Supabase Production** بملكيتك + نسخ احتياطي مفعّل.
2. **SQL: P0 + P1 + P2** مطبّقة ومتحقق منها.
3. **Vercel Production** على دومين ثابت (حتى `*.vercel.app` مقبول لPilot داخلي).
4. **مفاتيح env صحيحة** (Anon + Service Role) بدون تسرّب.
5. **Auth**: بريد/كلمة مرور يعمل؛ Google إن وعدت به في العقد.
6. **عزل المستأجر**: مستخدم المتجر لا يرى بيانات متجر آخر (اختبار يدوي إلزامي).
7. **Storage** للشعار/الصور.
8. **حساب owner** للمتجر التجريبي (ليس فقط demo عام).
9. **Sentry** (بشدة: أول عميل = أخطاء حقيقية في الميدان).
10. **سياسة نسخ احتياطي**: إما Supabase PITR/Pro أو تصدير دوري من شاشة Backup + جدول تشغيل.
11. **اختبار POS على جهاز لمس حقيقي** (تابلت/جوال كاشير) + طباعة إن وُعدت.
12. **اتفاقية بيانات بسيطة** (حتى صفحة خصوصية مختصرة) — خاصة في السعودية/اليمن للبيانات التجارية.

4. دفع الكود الحالي + وسم `v1.2.0-p2`.
5. حماية فرع `main` + فرع `develop`.

### المرحلة C — قواعد البيانات
6. إنشاء Supabase **`rafd-dev`**.
7. تنفيذ `p0_security.sql` → `p1_features.sql` → `p2_features.sql`.
8. إنشاء Storage bucket للوسائط + سياسات عامة للقراءة إن لزم.
9. ضبط Auth: Site URL، Redirect URLs لـ `localhost` و preview.
10. إنشاء مستخدمين: `owner` تجريبي + `superadmin` (كلمات مرور قوية غير `password123` في Prod لاحقاً).
11. بذر Dev: مستأجر، فرع، منتجات، حسابات بنكية يمنية تجريبية.

### المرحلة D — الربط المحلي
12. `.env.local` بمفاتيح Dev فقط.
13. `npm ci && npm test && npm run build`.
14. Smoke يدوي: Login → Dashboard → POS بيع → مخزون → رفع شعار → تقرير.

### المرحلة E — الاستضافة
15. إنشاء مشروع Vercel `rafd-app` وربطه بـ GitHub.
16. Env Preview = Dev Supabase؛ تعطيل Productioن مؤقتاً أو حمايته.
17. تحقق Preview deployment أخضر.

### المرحلة F — الإنتاج
18. إنشاء Supabase **`rafd-prod`**.
19. تنفيذ نفس SQL بالترتيب.
20. Storage + Auth URLs على دومين الإنتاج.
21. Vercel Production env = مفاتيح Prod + `SENTRY_DSN` إن جاهز.
22. Deploy من `main`.
23. Smoke على Production (نفس قائمة D) **بدون** بيانات وهمية مفرطة.

### المرحلة G — المراقبة والدومين
24. Sentry: تفعيل + تنبيه بريد.
25. (اختياري) Uptime على `/api/health`.
26. ربط دومين مخصص + تحديث Google OAuth و Supabase redirects.

### المرحلة H — Pilot (أول نسخة تجريبية)
27. إنشاء مستأجر حقيقي (متجر تجريبي) مع عملة YER.
28. تدريب 30–60 دقيقة: POS، آجل، مخزون كرتون.
29. تشغيل أسبوع مراقبة + قناة دعم واتساب بشرية.
30. بعدها فقط: قرار P3 أو تفعيل WhatsApp Cloud / دفع اشتراكات.

### بوابة خروج Beta (Checklist سريع)
- [ ] `npm test` و `npm run build` ينجحان على CI
- [ ] SQL P0–P2 على Prod
- [ ] Login + JWT على API
- [ ] عزل tenant
- [ ] POS نقدي + آجل + تحويل لحساب بنكي
- [ ] فاتورة طباعة/مشاركة
- [ ] Backup يدوياً مرة واحدة على الأقل
- [ ] Sentry يستقبل event تجريبي
- [ ] لا توجد مفاتيح sandbox في Production
